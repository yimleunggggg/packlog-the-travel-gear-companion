# PACKLOG 产品与运维说明（存档）

> 供产品/运营/负责人查阅；技术实现以仓库代码为准。

## 1. 重量「AI」与 HTTP 接口（大白话）

- 若在环境变量里配置了 **`VITE_WEIGHT_AI_URL`**，前端会向**你自己部署的一个小网址**发请求估算重量；网址背后可接 OpenAI/Gemini 等。**浏览器里不存放 AI 密钥。**
- **未配置或请求失败**时，使用**本地启发式**（按品类、名称里是否像「35L」等猜区间），**不联网**。
- 没有内置「全球巨型商品库」；装备数据主要来自种子数据与用户自建。

## 2. 网页联网与 CORS（大白话）

- 网页只能用 **`fetch` 访问你配置的地址**；对方服务器若未允许你的域名跨域（**CORS**），浏览器会拦截。
- 不能随意抓取任意网站当数据库；公开数据通常需**你自己的后端**整理后再给前端用。

## 3. 用户数据存在哪

- **默认**：浏览器 **`localStorage`** 存一整份快照（行程 + 装备库）。
- **可选云端**：设置 **`VITE_DATA_BACKEND=supabase`** 并配置 Supabase 后，写入表 **`packlog_snapshots`**（按 `workspace` 一条 JSON 快照）。
- **真正多用户、防串数据**需在 Supabase 里建表、配置 **RLS**，并把 `workspace` 与用户 ID 绑定规则定清楚。

## 4. 登录与营销订阅（实现要点）

- 登录走 **Supabase Auth**（魔法链接 / Google）。
- **产品更新邮件**：登录半屏内提供可选勾选框；写入 **`user.raw_user_meta_data.marketing_opt_in`**（boolean）。魔法链接通过 `signInWithOtp` 的 `data` 传入；Google 在跳转前写入短时 **localStorage** 意向，回调成功后 **`updateUser`** 合并（避免 OAuth 丢字段）。
- 后续可在 Supabase SQL 里筛选 `(raw_user_meta_data->>'marketing_opt_in')::boolean = true` 导出邮箱列表（配合合规与退订流程）。

## 5. Supabase 你需要动手做的事（清单）

1. 创建项目，拿到 **URL** 与 **anon key**。
2. 在库里建 **`packlog_snapshots`**（若用云端快照）及文档中的其它表。
3. **Authentication**：开启邮箱/Google；**Redirect URLs** 包含本地与线上的 **`/auth/callback`**。
4. **RLS**：每张表配置「只能操作自己的行」等策略。
5. 部署环境写入 **`VITE_SUPABASE_*`** 等变量。

**同步逻辑**（离线优先还是云端优先、冲突谁赢）需产品与研发共同定规则后写代码，不是控制台单独能完成的。

## 6. 「社区中位数」为何默认不是「全网」

- 当前实现只在**本机已加载的** `library + trips` 里凑样本算中位数。
- **全网中位数**需要用户样本**汇总到服务端表**再聚合，属单独一期（含隐私与合规）。

## 7. 已实现 vs 待排期（概括）

**已实现（概括）**：行前装备主流程、目的地搜索/级联、默认三箱型、社区页与克隆、公开/标签、重量分层与可选 AI URL、延迟登录与移动端适配、本地快照（及可选 Supabase 快照路径）。

**待排期/待讨论**：国内访问质量与部署选型、完整多表同步与冲突策略、管理员后台、全网重量样本、巨型商品库、邮件发送管道与退订页等。
